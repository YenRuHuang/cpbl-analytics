Rebas 野球革命
改善建議

以訂閱會員 + 每天使用 Baseball Savant / FanGraphs 的角度，
針對效能、SEO、手機體驗、功能、無障礙提出具體可行的建議。

分析者：Murs | 身份：Rebas 訂閱會員 | 日期：2026-04-05

⚡ 效能優化

這些改動不需要動業務邏輯，純工程面優化，可以很快看到效果。

圖片沒有 Lazy Loading

影響大

首頁有 166 張圖片，但沒有任何一張設定 loading="lazy"。使用者一進頁面就同時下載所有圖片，包括在畫面外看不到的。

建議做法

在首屏以下的 <img> 標籤加上 loading="lazy"。瀏覽器會自動延遲載入看不到的圖片。改動量很小，效果很明顯 — 初始載入的請求數可以減少一半以上。

中文字體分片過多

影響大

使用 Google Fonts 的 Noto Sans TC，單一頁面載入了 18 個 woff2 分片。中文字體因為字元量大被切成很多 chunk，每個 chunk 都是一個 HTTP 請求。

建議做法

自己 subset 常用的 3000-4000 個漢字（涵蓋球員姓名 + 常用介面文字），打成 1-2 個 woff2 檔 self-host。搭配 font-display: swap，文字不會因為字體載入而閃爍。

圖片格式沒有用 WebP

中等影響

全站圖片以 PNG（52 張）和 JPG（22 張）為主，沒有使用 WebP 或 AVIF。WebP 在同樣品質下體積比 PNG 小 25-35%。

建議做法

用 <picture> 標籤搭配 WebP 和原圖 fallback。如果用 CDN（如 Cloudflare），很多可以自動做圖片格式轉換。

缺少 Resource Hints

中等影響

沒有 preconnect 或 preload 標籤。每次載入 Google Fonts 或第三方資源都要重新做 DNS lookup + TLS handshake。

建議做法

在 <head> 加入：
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
可以省下 100-200ms。

🔎 SEO 改善

球員頁面和排行榜是最有搜尋價值的內容，但目前可能沒被好好索引。

SPA 架構不利搜尋引擎

影響大

整站是 Create React App 的 CSR（Client-Side Rendering），HTML 只有一個空的 <div id="root">，所有內容都靠 JavaScript 在瀏覽器端 render。Google 爬蟲雖然可以跑 JS，但索引效果遠不如 SSR，而且爬蟲的 rendering budget 有限。

建議做法

長期：考慮遷移到 Next.js 做 SSR / ISR（Incremental Static Regeneration）。排行榜和球員頁面特別適合 ISR — 資料每天更新一次，不需要即時 SSR。

短期：至少做 pre-rendering，把 Google 容易爬到的靜態 HTML 先生出來。

缺少 Canonical 和結構化資料

影響大

沒有 <link rel="canonical">（可能有重複內容問題，尤其 www / stats / blog 多個子網域），也沒有 JSON-LD 結構化資料。

建議做法

每頁加上 canonical 標籤。球員頁面加 schema.org/Person JSON-LD，比賽頁面加 schema.org/SportsEvent。這些能讓 Google 搜尋結果出現 rich snippet — 搜「張育成 數據」就有機會直接看到你們的卡片。

子網域分散 SEO 權重

中等影響

目前有 www.rebas.tw、stats.rebas.tw、blog.rebas.tw、account.rebas.tw 四個子網域。搜尋引擎會把子網域當成不同網站，SEO 權重分散。考慮整合為路徑（/stats、/blog），讓所有 backlink 都累積在同一個 domain 上。

📱 手機版體驗

棒球迷很多是在手機上看數據的，手機體驗直接影響留存。

排行榜表格在手機上不可讀

影響大

進階數據排行榜有 17+ 個欄位，在 390px 寬的手機上全部壓縮在一起，數字幾乎看不清楚。

建議做法

固定左邊的排名和球員名欄位（sticky column），其他欄位水平滾動。FanGraphs 就是這樣處理的。

或者提供「精簡模式」，手機版預設只顯示 5-6 個最重要的欄位（OPS+、wOBA、K%、BB%、AVG），要看完整的再展開。

球員頁上半部過於擁擠

中等影響

PR 表、落點圖、近三年成績在手機上全部擠在同一區。改成垂直堆疊 + 可收合面板（accordion）會好很多，讓使用者自己選要看什麼。

導航列沒有 Hamburger Menu

中等影響

在窄螢幕上，導航列的文字只是縮小而不是收合成漢堡選單。「分析工具」和「新數據」的下拉式子選單在手機上很難點到。

💡 功能建議

參考 MLB 生態系的做法，這些功能可以提升用戶黏著度。

球員比較功能

參考 FanGraphs

FanGraphs 有比較工具可以把兩個球員的數據疊在一起看。你們目前只能看單一球員頁面。如果可以做 side-by-side 的 PR 表和 Rzone 比較，球迷在討論「張育成跟林安可誰比較強」的時候就會直接貼你們的連結。

這對社群分享和自然流量都很有幫助。

篩選條件 Deep Link

參考 Savant

Baseball Savant 的搜尋器做得很好 — 篩選條件都會寫進 URL，可以直接把一個複雜的查詢結果分享給朋友。

你們 Rzone 的篩選功能很強（對手、球數、壘包、打擊結果都有），但篩完之後 URL 不會變。沒辦法把「張育成面對獅隊兩好球之後的熱區圖」分享出去。把篩選狀態 sync 到 URL query params 就可以解決。

Statcast 風格搜尋器

參考 Savant

長期可以考慮做一個類似 Savant Statcast Search 的多條件交叉查詢工具。讓進階用戶可以自訂查詢條件（球種 + 球數 + 進壘區域 + 打擊結果 + …），你們的逐球資料粒度已經夠了，只是缺一個前端介面。

這會是「新手專區」的反面 — 給重度數據迷用的進階工具，也是訂閱制的賣點。

排行榜欄位可排序

參考 FanGraphs

FanGraphs 的排行榜每個欄位都可以點擊排序。你們目前似乎是固定排序。讓用戶自己決定要按 wOBA、K% 還是 Whiff% 排，基本功能但使用頻率很高。

人工標記 ML 輔助

長期方向

從「專屬功能」頁可以看到每場比賽的逐球球種 / 進壘點都需要人工標記。2026 球季已完成 13 場，同時也在徵 4 名影像紀錄員。

如果用歷史已標記的資料訓練分類器（投手慣用球種配比 + 球速區間 → 球種預測），就可以做 ML 預標記，標記人員只需要校正模型判斷錯的部分。特別是球種辨識這種有明確 pattern 的任務，準確率應該可以到 85%+，等於省掉大部分的重複工作。

🛠 技術架構觀察

從前端 bundle 和 network requests 觀察到的技術棧現況。

前端框架 Create React App（Webpack bundle，CSR）
SSR / SSG 無 — 純 Client-Side Rendering
Service Worker 無 — 沒有離線快取或推播能力
API 格式 RESTful GET（如 /api/seasons/{id}/leaders）
第三方服務 Google Tag Manager、Facebook Pixel、Cloudflare Insights
已知 API 小問題 排行榜請求帶 pa=undefined 參數
架構升級建議
  • • 考慮遷移到 Next.js（SSR / ISR），一次解決 SEO、首屏速度、code splitting
  • • 加入 Service Worker，支援離線快取和推播通知（新數據更新完成時通知訂閱用戶）
  • • API 層加 Cache-Control header，排行榜等不常變動的資料不需要每次都打 DB
  • • 子網域整合為路徑，集中 SEO 權重

🚀 即將推出功能的想法

你們方案頁提到的 roadmap，我有一些實作上的想法。

打者 wRC+

你們已經有 wOBA 了，wRC+ 的公式本身不難。比較麻煩的是 Park Factor — CPBL 五個主場的打擊環境差蠻多的，洲際跟大巨蛋的得分環境完全不同。

要考慮的問題：Park Factor 用幾年的資料算？球場整修後資料怎麼處理？客場比賽只佔一半，樣本量夠不夠穩定？我有用自己 DB 裡的 2025 全季資料試算過一版，踩過一些坑可以分享。

聯盟平均本壘板紀律

你們球員 Rzone 已經有核心區 / 邊線區 / 追打區 / 無效區的分區，也有標示聯盟平均的欄位。做聯盟平均要注意不能只算一個「大平均」— 左打右打要分開、球數情境也要分開。打者領先時的追打率跟打者落後時差很多，混在一起就沒有參考意義了。

打者 WAR

你們目前只有投手 WAR（tRA-based），打者 WAR 更複雜：Batting Runs + Baserunning Runs + Fielding Runs + 守位調整 + 聯盟調整 + 替補水準。

最難的是守備評價 — 沒有 Statcast 的情況下，可以用 RF/9 + 你們已有的守備指標做近似。FanGraphs 早期也是用比較粗的守備模型在算 fWAR，CPBL 用類似的方法也完全合理。重點是把方法論說清楚，讓使用者知道這個 WAR 的侷限。
